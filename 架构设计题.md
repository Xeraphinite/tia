### 模块一：Context / Performance

2. 一个 session 连续聊了 200 轮，context 快爆了。你会怎么做压缩？如何确保压缩后的对话仍然流畅？

上下文按照使用的类型而言可以分成四块：永远保留的规则和用户限制；当前目标、决定、待办等工作状态；最近十几轮原文；较早对话的分层摘要。考虑到模型常常在使用至 75% 左右的时候会出现能力下降，因此在这个时候应当触发压缩。摘要保留人物、时间、事实、决定、未完成事项等内容。涉及准确数字或历史细节时，再从原始聊天中检索。每隔几次压缩，根据原始聊天重新生成摘要，减少摘要不断改写造成的偏差。

### 模块二：Memory

3. 半个月后用户问了以前问过的问题，怎么召回

首先需要判断用户是在寻找之前的答案、继续之前的话题，还是重新询问一个相似问题。因此 Memory 中除了保存提取后的用户偏好和事实，还应当保留原始对话、问题与答案的对应关系、时间以及所属 Session。

召回时可以先根据关键词、实体和时间做精确检索，再通过向量检索寻找语义相似的问题，最后结合相似度、时间、重要性和用户后续反馈进行重排。召回的内容应当包含原始问题、当时的回答和附近几轮对话，避免一条脱离上下文的记忆造成误解。如果第一次召回的信息不足，还可以让模型分析缺少哪些背景，再进行一次补充检索。

对于用户偏好、个人经历等相对稳定的信息，可以直接使用之前的记忆。对于价格、政策、新闻、软件版本等容易变化的信息，之前的对话只能证明用户问过这个问题，答案仍然需要根据当前情况重新生成。如果新旧记忆发生冲突，应当保留时间和来源，并优先采用用户最新的明确表达。召回置信度较低时，可以告诉用户以前似乎讨论过类似问题，避免错误地声称自己记得。

### 模块三：Task

5. 长程任务执行中忘记目标，怎么解决

模型很容易在运行的过程中产生语意偏移的问题，因此对于长程问题而言，目标和进度必须放在模型的 Context 外部。可以考虑在外部维护一个结构化的任务状态，大致包含了现状以及目标。对于目标部分，这个结构化的状态应当包括约束、验收标准等，而现状部分则应当包括当前计划、待办、下一步和阻塞项等。在长程任务中，模型每次执行前都需要读取这个结构化状态，并在执行后更新它。这样可以避免模型忘记目标或偏离目标。考虑到模型可能会出现虚假完成，因此在任务完成时需要有独立的外部的验证机制，例如测试、交付的结果，甚至可以使用一个 subagent 进行验证（当然 Prompt 或者 rule 还是需要设定好的）。

### 模块四：Tool / Session Runtime

## 7. 异步工具怎么执行和通知

对于异步工具而言，一次调用应当被抽象成一个独立的 Job。Agent 发起调用后，Runtime 立即返回 Job ID、当前状态、预计完成时间以及取消入口，Session 可以继续响应用户，无需一直保持 busy。

Job 的状态可以包括排队、运行、等待外部系统、成功、失败和取消。后台执行器通过持久化队列领取任务，并定期写入进度和心跳。对于可能重试的任务，需要设置幂等键，避免重复发送消息、重复创建资源或重复扣款。执行结果较大时，可以保存到 Artifact Store，Job 中只记录结果摘要和访问地址。工具完成后，Runtime 先将结果事件写入 Session 的事件日志，再决定后续行为。如果当前任务正在等待这个工具，Runtime 可以携带工具结果创建一个新的 continuation run，让 Agent 继续推理。如果结果本身已经可以直接交付，就向用户发送结果卡片。如果用户已经取消任务或修改了目标，则保存结果并通知用户，由用户决定是否继续使用。

工具完成和用户通知应当分别记录状态，因为工具可能已经成功，通知仍然可能发送失败。通知可以通过应用内消息、推送或邮件完成，并支持离线补发。考虑到事件系统通常采用至少一次投递，因此消费者还需要根据 Job ID 和事件 ID 去重。


# 模块五：Agent Runtime 架构对比

## 9. Claude Code 与 GLM、豆包等 Function Calling 有什么区别

这个问题需要区分模型调用协议和完整的 Agent Runtime。Claude Code 除了使用模型的 Tool Use 协议，还提供了持续存在的 Shell、文件系统、权限控制、后台任务和工具输出管理。GLM、豆包等 OpenAI Compatible 接口主要规定模型如何申请调用工具，具体的执行环境需要应用自行实现。在 Anthropic 的协议中，Claude 通过 tool_use 内容块发起调用，Runtime 执行工具后，通过对应 tool_use_id 的 tool_result 内容块返回结果。工具结果可以包含文本、图片、文档和明确的错误状态。在 Claude Code 中，过长的工具输出还可以写入文件，只把摘要和文件位置放回 Context，从而减少工具日志对上下文的占用。

OpenAI Compatible Function Calling 通常由模型在 Assistant 消息中返回 tool_calls，其中包含工具名称和 JSON 参数。应用执行工具后，再通过带有 tool_call_id 的 role=tool 消息返回结果。GLM 和豆包大体兼容这一形式，因此接入现有 SDK、网关和 Agent 框架相对方便。Claude 的方式对富媒体结果、错误信息和多个有序工具调用表达得更完整，也和 Claude Code 的执行环境结合得更紧密，但消息顺序和格式要求更严格。OpenAI Compatible 方式结构简单、生态成熟、模型替换方便，但异步任务、进度通知、结果文件、取消和恢复等能力需要 Runtime 自行补充。

在实际架构中，可以在内部统一定义工具请求、执行进度、执行成功和执行失败等事件，再分别转换成 Anthropic 或 OpenAI Compatible 格式。这样 Session 和 Tool Runtime 不会与某一家模型协议绑定。
